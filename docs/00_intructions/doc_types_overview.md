# ドキュメント種別概要

- 命名規則・フォルダ構成は「00_naming_and_structure_reference.md」を参照。

---

## 1. 設計文書・画面概要

- **画面概要 (Screen Overview):**
  - モデル、フロー、UI/UXの概要、各画面の主な処理の説明。
  - 画面の目的、機能、操作フローを理解するための情報。
- **画面定義書 (Screen Definition):**
  - 画面の詳細なレイアウト、フィールド、イベント、処理の定義。
  - UIコード生成、業務ルールの検証、データフローの確認に使用。
- **画面イメージ/Mock (Screen Mock):**
  - HTML/CSS/画像ファイルのサンプル、`docs/03_画面系/00_画面イメージ/`に保存。
  - UI/UXの標準化、開発者とレビューに使用。

## 2. API・データ文書

- **API一覧 (API List):**
  - APIエンドポイント、機能、画面との関連。
  - フロントエンドとバックエンドの接続点を特定。
- **API定義書 (API Definition):**
  - エンドポイント、リクエスト/レスポンス、エラー、業務との関連。
  - APIコード生成、統合テストに使用。
- **テーブル定義書 (Table Definition):**
  - データベースの構造、フィールド、データ型、制約。
  - DB設計、モデル、マイグレーションに使用。

## 3. プロセス・運用文書

- **gen_process_jp.md:**
  - 日本のSIの標準プロセス、AIの役割、開発・レビュー・運用のステップ。
  - チームが標準プロセスに従い、品質と顧客との協力に使用。
- **copilot-instructions.md:**
  - AIエージェントに向けたアーキテクチャ、ワークフロー、コードベースのルール。
  - AIが生成したコードが正しく、プロジェクトと同期に使用。
- **nextjs_pos_base_setup.md:**
  - ソースベースの再構築、構造、設定、プロジェクトの初期化のステップ。
  - 新しい開発者またはAIエージェントが環境を迅速に構築に使用。
- **README.md:**
  - プロジェクトの概要、実行方法、参考資料。
  - プロジェクトのエントリポイント。

## 4. テスト・運用文書

- **Test case/仕様書:**
  - テストケース、入力/出力条件、検証が必要な業務の定義。
  - 品質、機能の確認前に実施。
- **運用マニュアル (Operation Manual):**
  - 運用の手順、障害対応、バックアップ/ロールバック、FAQ。
  - 実運用のサポート、展開時のリスク削減。

---

## 全体的な意義

- 各種ドキュメントは、開発者、顧客、AIエージェントの間で品質と標準化を保証。
- ドキュメントは、コード生成、テスト、レビュー、運用、拡張に使用。
- ドキュメントの維持は、SI、AI、複雑な業務システムの成功に不可欠。
